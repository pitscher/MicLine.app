## §9 M2–M5 playbooks

Each milestone starts the same way: **re-run the interview, then the loop.** Nothing else changes. And to say it once, plainly: **execution order is M1 → M2 → M4 → M5 → M3-if-ever.** Monetization keeps its historical number but runs *last by design* — MicLine is fully live, polished, and boringly operable before you even revisit whether to charge for it.

**Milestone kickoff ritual (all of M2–M5):**
1. `[MODEL: strongest available] [EFFORT: max] [SESSION: fresh]` →
   `Read docs/product/epics/⟨M⟩-*.md, feature-inventory.md, decision-log.md, and the current codebase structure. Interview me (P1 rules: batches ≤5, be critical, challenge scope) until every feature in this epic is defined to the same depth as an M1 feature brief. Write/refresh docs/product/feature-briefs/ for each, update the inventory + milestone map, and list which "Demands on M1" were actually satisfied vs. need remediation features.`
2. 🛑 Gate the briefs, commit, then run the [file 05](05-feature-loop.md) loop per feature in dependency order. When a milestone's user-visible scope is complete, cut its Release per the semver policy in [file 03 §D](03-github-operations.md) (e.g. M2 done → `v2.0.0`).

### 9.1 M2 — Admin interface + entitlements/vouchers + stand-out features
- **Sequence:** entitlement engine first (it's the substrate), then admin UI, then vouchers *inside* the admin UI, then the stand-out features (many will themselves be gated capabilities).
- Entitlement engine feature: the Article-VI capability interface becomes real — capability enum in `packages/shared`; `grants` table (D1) with source (`baseline|voucher|admin|billing-future`), scope (per-user / per-event), expiry, revocation; a single `can(env, actor, capability, ctx)` service; audit-log append on every grant/revoke. **Your voucher sketch (the P1 topic-E answers) goes verbatim into the kickoff interview** — the interview's job is to pin: gated dimensions (features vs. event count vs. attendee cap vs. duration), single-use vs. multi-use codes, redemption timing (at event creation), expiry/revocation, and the ungated free baseline.
- Admin UI: `/admin` route tree behind `requireAdmin()` (the F001 `role` column pays off); MUST cover everything you'd otherwise do in the Cloudflare dashboard or D1 console — user/event search + detail, force-end/soft-delete event, voucher generate/list/revoke, grant inspector, abuse queue, headline metrics, a **usage & cost panel** (requests, DO minutes, D1 rows, KV ops, emails sent → projected € via the M5 cost model — your "map usage to money" wish lives here), and a GDPR deletion action (Article V). Its spec's clarify step is where you enumerate every "I never want to touch the DB for this" scenario.
- Stand-out features: implement in the force-ranked order from the P1 brainstorm; each is a normal file-05 loop — and each ships in whatever MINOR release you cut next; features don't wait for the whole milestone. One ready-made candidate for the brainstorm's keep/kill round: a **translation pipeline** — connect a free-for-open-source TMS (Weblate or Crowdin) that syncs with GitHub via PRs touching only the locale catalogs (the CONTRIBUTING rule already anticipates this). That *is* the "translation dashboard updated by GitHub automation" — rented, not built; if you ran the optional Lunaria loop from [file 06](06-m1-plan.md), the TMS either replaces it or feeds it — decide in the interview.

### 9.2 M3 — Monetization (PARKED)
Do nothing until you decide to earn money. When you do: kickoff ritual on `M3-monetization.md`; billing provider (Stripe or a **Merchant-of-Record** like Paddle/Lemon Squeezy — an MoR handles EU VAT for you, which materially reduces the German paperwork; weigh in the interview) integrates as *one new grantor* writing rows into the M2 `grants` table — if any other code needs changing, M2 violated Article VI and converge/fix that first. Resolve the parked German-operator checklist (Impressum, USt/Kleinunternehmer, AGB, Widerruf, invoicing) with a Steuerberater/lawyer before charging anyone — that's a legal workstream, not a coding one.

### 9.3 M4 — UI overhaul (landing · moderator console · audience board · admin)
Designing good UI genuinely is the hard part, so M4 gets a concrete four-step workflow instead of vibes:
1. **Design brief** — the kickoff interview here is a design interview (brand adjectives, 3–5 reference sites, typography/color direction, motion tolerance), building on the P1 topic-F notes. Output: `docs/product/design-brief.md`. Content/copy is part of it: each surface's respec will include its copy per locale — Claude drafts, you gate.
2. **Check the tooling *then*** — and lead with **Claude Design** (correcting an earlier draft of this guide that predated it): Anthropic Labs' visual-design product, in research preview since April 2026, is **included with your Pro plan** and, since its June 2026 update, imports/enforces design systems, round-trips with Claude Code (`/design` and `/design-sync` commands in the terminal), and exports PPTX/PDF/Canva. Two Pro-plan cautions verified as of 2026-07-05: it **shares your usage limits** with chat and Claude Code (budget sessions like any other model work), and it's a preview product that has already changed substantially once — re-verify capabilities at M4 time. Also open `/plugin` → Discover for design skills; keep anything only if it earns its context cost.
3. **Design system first** — one loop producing `docs/product/design-system.md` + the real `packages/ui` tokens/components. Claude Design's design-system import (or a `/design-sync` from a first exploration) may generate the first cut — gate it like any artifact. Every surface respec then references it, so the four surfaces can't drift apart.
4. **Per surface: the screenshot loop** — one surface = one [file 05](05-feature-loop.md) feature; because M1/M2 kept logic out of components, these loops touch only presentation. Iterate visually — in Claude Design's canvas where it helps, and always in the running app: `pnpm dev` → ⌘⇧4 screenshot → annotate in Preview/Markup → paste into Claude Code with change requests → repeat until the brief's adjectives hold on your phone *and* a laptop. Checklist step mandatory (a11y + responsive audit) before each surface merges.

### 9.4 M5 — Longevity (perf, cost, stability, ops)
- Start with measurement, not opinions: a loop for observability (structured logs → Workers Logs/Logpush decision, DO/D1/KV usage dashboards, cost report) — you can't refactor for cost you can't see.
- Then: `/speckit.converge`-style audit of the whole repo against the constitution (`Acting per Article III, audit the codebase for operator burden: dependency count/freshness, dead code, per-event DO storage growth, missing retention enforcement, alarm/cron cleanup jobs. Output a ranked refactor backlog.`) → each accepted item becomes a small file-05 loop.
- Implement the event data-retention policy decided in P4 (DO cleanup via alarms, event_stats as the durable record).
- Load-test the DO path (scripted WS clients; e.g. 500 participants, 1 event) before declaring stability.
- Author the **rebuild-from-zero runbook** (`docs/runbooks/rebuild-from-zero.md`) — your requested emergency document: a loop whose spec is literally "starting from nothing but this repo, reach a live PRD". Its raw material is the `docs/runbooks/manual-config-log.md` you've kept since [file 02](02-setup.md) plus F000's `resource-bootstrap.md`; Claude drafts the runbook from log + workflows + wrangler config, and if you want it *proven*, walk it once against a scratch Cloudflare account. This is also the moment the Terraform/Pulumi question re-opens **with data** ([file 02 §2.8](02-setup.md) posture): count the runbook's remaining console-only lines — a handful means the posture holds; a page means evaluate the Cloudflare providers now, against their then-current coverage.
- Produce the **cost model** (`docs/ops/cost-model.md`): per-event and per-month formulas from *measured* usage (requests, DO duration, D1 row reads/writes, KV ops, emails) × current Cloudflare prices, with 2–3 worked scenarios (a 50-person meetup, a 500-person conference day, a slow month). This feeds the M2 admin cost panel and turns "what does growth cost me" into arithmetic.
