## §9 M5–M13 playbooks

Each milestone starts the same way: **re-run the interview, then the loop.** Nothing else changes. Execution order is **sequential — M5 → M6 → M7 → M8 → M9 → M10 → M11 → M12**, each completing as its target release (`v0.5.0` … `v0.11.0`, then **`v1.0.0` at M12**); **M13 runs only if ever**, as provisional `v1.1.0`. The release gates along the way (feature inventory, "Release gates"): after M8 MicLine is an *operationally controlled public service*; after M9 a *differentiated product*; after M12 the *first production-mature release*. Monetization is deliberately last — MicLine is fully live, polished, and boringly operable before you even revisit whether to charge for it.

**Milestone kickoff ritual (all of M5–M13):**
1. `[MODEL: strongest available] [EFFORT: max] [SESSION: fresh]` →
   `Read docs/product/epics/⟨M⟩-*.md, feature-inventory.md, decision-log.md (this milestone's rows + its "Scheduled decisions" entries), and the current codebase structure. Interview me (P1 rules: batches ≤5, be critical, challenge scope) until every feature in this epic is defined to the same depth as an M1–M4 feature brief, and every kickoff-owned decision from the decision log's "Scheduled decisions" registry is resolved or explicitly re-owned. Write/refresh docs/product/feature-briefs/ for each, update the feature inventory, and list which "Demands on M1–M4" were actually satisfied vs. need remediation features.`
2. 🛑 Gate the briefs, commit, then run the [file 05](05-feature-loop.md) loop per feature in dependency order. When the milestone's scope is complete, cut its target release per the SemVer policy in [file 03 §D](03-github-operations.md) (e.g. M5 done → `v0.5.0`); mid-milestone increments you want on PRD early ship as prereleases of that target (`v0.5.0-beta.1`).

### 9.1 M5 — Moderator control and data portability (`v0.5.0`)
- **Scope (inventory):** recap & export (STD-05 — rank 1, raised in P2: every event before export ships loses its content on day 3), intake close (MOD-05), hide board (MOD-06), regenerate board link (MOD-07), follow-up flag (MOD-08).
- **Export is the anchor:** one-click CSV + JSON + Line snapshot (HTML), no export-time filters — the export is everything (questions, attribution, times, final statuses incl. declined + reasons, merge structure, votes, follow-up flags, primer, stats; the curated-mode columns stay empty until M9 exists). Build it as a **service, not a UI one-off** — the M7 emergency machinery consumes the same snapshot builder for its E-4b exports. Responsibility line at export time and on the privacy page (decision T7-3); entitlement-independent, never gated.
- **Kickoff owns (decision log, "Scheduled decisions"):** the final export-format list and the final user-facing name for "Line snapshot (HTML)".
- **Live controls cluster:** intake close (manual toggle only, freely reversible — no timers, REJ-07), hide board (board-only; neutral customizable message at hide-time; no QR while hidden; ops logs only, never audit-logged — T3-5), regenerate board link (leak recovery for the capability URL — the adopted alternative to the rejected board password, T8-8). Follow-up flag: moderator-only boolean, export column, purge-bound.

### 9.2 M6 — Operator control plane (`v0.6.0`)
- **Scope (inventory):** admin bootstrap (ADM-01 — role exclusively via the `ADMIN_EMAILS` Wrangler secret; recovery = update the secret), the dedicated admin auth wall (ADM-02 — mechanism per the tech interview's OQ1 decision; hard requirement: emergency reset always possible via Cloudflare account access alone; step-up confirmation on destructive actions regardless), oversight & abuse tools (ADM-03/04), metrics wall (ADM-05), global stats (ADM-06), platform controls (ADM-07 — the admin UI wraps the M1 KV flags), feedback-email setting (ADM-10, takes over from the M4 secret), consequence previews & reversal matrix (ADM-11).
- `/admin` route tree behind `requireAdmin()` + the dedicated wall (the M1 `role` column pays off). It MUST cover everything you'd otherwise do in the Cloudflare dashboard or D1 console — search events/moderators + detail, read-only god view of live events (incl. the M5 live states: board hidden, intake closed), force-end (calm outside, truthful inside, email E-3), soft-delete ("removed by administration"), ban moderator email (silent), GDPR full delete, headline metrics, and a **usage & cost panel** (requests, DO minutes, D1 rows, KV ops, emails sent — the projected € arrives with the M12 cost model; your "map usage to money" wish lives here). Its spec's clarify step is where you enumerate every "I never want to touch the DB for this" scenario.
- **ADM-11 is a design rule, not a feature:** every admin control shows a quantified consequence preview ("X events: X running, X scheduled, X in export window") with a computation timestamp, and defines its reverse action in the UI; nothing requires a redeploy to undo; irreversible parts labeled honestly. E-3 (force-end notice) is the milestone's email; wording at the M6 spec.

### 9.3 M7 — Emergency operations and recovery (`v0.7.0`)
- **Scope (inventory):** emergency stop (ADM-08) and delete-all-data (ADM-09) — two separate controls, never bundled (decision D8/T8-6).
- **Emergency stop = total stop:** freezes events, participant surfaces, boards, console, moderator auth, creation, registration, and all scheduled jobs including purges, with exactly three exceptions — static notice pages, the admin UI (reversal must never require a redeploy), the expiring export-download endpoint. Sequence: freeze instantly → snapshot exports for affected events (the M5 export service) → emails E-4 then E-4b; E-5 on reversal. Trigger modal: export choice default ON with "skip if you suspect data compromise", quantified impact preview, computation timestamp, typed confirmation + step-up. Scheduled events never open during a stop; on reversal they open late if their window is current, else auto-end; overdue purges run 24 h after resume. Participants/boards see the calm emergency-ended screen — the sole participant-visible platform-initiated ending.
- **Delete-all-data:** the separate GDPR/breach nuke — step-up auth, typed confirmation, honest irreversibility labeling (D1 time-travel ≤30 days is an infra safety net; DO storage is not); operationally preceded by an emergency stop (runbook).
- **M7 spec owns (decision log):** the emergency copy set (E-4/E-4b/E-5, emergency-ended screens), trigger-modal wording, export-download endpoint isolation, scheduled-event resumption copy, delete-all-data runbook linkage. The M4 purge jobs gain their pause/resume-grace behavior here (EVT-07/OPS-03 extensions).

### 9.4 M8 — Entitlements, invite codes and lockdown (`v0.8.0`)
- **Sequence: entitlement engine first (it's the substrate), then vouchers inside the admin UI, then lockdown.** The Article-VI capability interface becomes real: capability enum in `packages/shared`; `grants` table (D1) with source (`baseline|invite-code|admin|billing-future`), scope, expiry, revocation; a single `can(env, actor, capability, ctx)` service; audit-log append on every grant/revoke.
- **Free baseline (lockdown off):** every shipped feature free; limits 100 participants/event (counted as concurrent presence, FR-2), 1 active + 1 planned event — cap enforcement begins here; the earlier beta window without it is an accepted risk (FR-1). **Never-gated:** core Q&A mechanics + all P2 moderator tools (intake close, hide board, regenerate board link, pool settings/limits, decline reasons, follow-up flag, filters/search/bulk, recap/export). **Gateable:** co-moderators, event branding, vanity slugs, limit raises, registration permission.
- **Your voucher decisions (decision log E1–E9 + T8-2/T8-3) go verbatim into the kickoff interview**, which resolves the scheduled OQ3 details: code format, QR sharing, custom message, redemption display, generation UX, redeem-by/grant-validity defaults, god-voucher presentation. Semantics already fixed: codes are bundles `{may_register? + grants + limit raises + validity}` redeemed into a wallet; two independent clocks; revocation hits the *next gated act*, never a running event; **rescheduling a committed event is never gated**; duplicate-event is creation and therefore gated.
- **Lockdown (invite-only mode):** gates new registration and new event creation; existing events run and start normally; doubles as the cost brake. The M6 platform-controls UI gains the toggle (ADM-07 extension); AUTH-04's non-empty-wallet purge carve-out activates. Release gate: **operationally controlled public service.**

### 9.5 M9 — Curated moderation (`v0.9.0`)
- **Scope (inventory):** curated line mode (STD-01 — pool → select/order/drag-reorder, return-to-pool, duplicate merge with primary + ×N badge, per-event submission limits, pool-visibility setting), upvotes + pool sort (STD-02 — votes advise the human, never reorder the line), review mode (STD-03 — open+review appends at approval time; curated+review releases into the pool; declined = owner-visible chip + optional reason), co-moderators (STD-04 — gateable, link-based invitations), console filters/search/bulk verbs (MOD-09 — exactly two bulk verbs: merge, bulk decline), per-viewer pool collapse (PRT-09).
- The F003 domain model was built **two-mode-shaped for exactly this milestone** — if curated mode needs a schema rewrite, M3 violated decision B10; converge that first. Curated ordering gets its reorder machinery here (fractional index or equivalent per tech-context).
- **Kickoff owns (decision log):** the cross-surface filter matrix incl. filter-by-submitter (guardrail: the full line is always the default view; the board keeps one canonical layout), curated defaults (pool visible, upvotes on — assumed, confirm), the per-participant submission-limit default/range, pool-collapse defaults (direction: collapsed on the board), and the co-moderator link-invite mechanism incl. link revocation and a co-moderator powers matrix (FR-5). Release gate: **differentiated MicLine product.**

### 9.6 M10 — Extended event toolkit (`v0.10.0`)
- **Scope (inventory):** speaker timer (STD-06), duplicate event → templates (STD-07 — a creation act, gated under lockdown), markdown-lite primer + announcements & the re-openable in-event primer (STD-08 — structure only, links stay inert, no images), vanity join slugs (STD-09 — gateable), rented TMS pipeline (STD-11), `de (formal)` event language (I18N-01).
- **The TMS decision:** connect a free-for-open-source TMS (Weblate or Crowdin) syncing catalogs with GitHub via PRs touching only the locale files (the CONTRIBUTING rule already anticipates this). That is the "translation dashboard, rented not built" — at the kickoff, decide whether it **replaces or feeds** the M4 Lunaria loop ([file 06 §8.4](06-m1-to-m4-plan.md)).
- **Kickoff owns (decision log):** the markdown-lite subset, the TMS replaces-or-feeds call (OQ4), and the formal-German register-variant implementation (device `de` + event `de (formal)` → direction: the event's variant wins when bases match).

### 9.7 M11 — Design system, identity and experience overhaul (`v0.11.0`)
Designing good UI genuinely is the hard part, so M11 gets a concrete four-step workflow instead of vibes:
1. **Design brief** — the kickoff interview here is a design interview (brand adjectives, 3–5 reference sites, typography/color direction, motion tolerance), building on the captured direction in the product brief (Luma north star, calm/clear/even-handed adjectives, dark-first, motion-explains-state, invisible host) — plus the two decisions the kickoff owns: **logo & color identity (OQ9, greenfield)** and the **board sound cue build decision (DSN-05** — moderator-configured, board-side mute, MicLine-provided sound set, browser autoplay-unlock constraint**)**. Output: `docs/product/design-brief.md` (DSN-01). Content/copy is part of it: each surface's respec includes its copy per locale — Claude drafts, you gate.
2. **Check the tooling *then*** — and lead with **Claude Design** (correcting an earlier draft of this guide that predated it): Anthropic Labs' visual-design product, in research preview since April 2026, is **included with your Pro plan** and, since its June 2026 update, imports/enforces design systems, round-trips with Claude Code (`/design` and `/design-sync` commands in the terminal), and exports PPTX/PDF/Canva. Two Pro-plan cautions verified as of 2026-07-05: it **shares your usage limits** with chat and Claude Code (budget sessions like any other model work), and it's a preview product that has already changed substantially once — re-verify capabilities at M11 time. Also open `/plugin` → Discover for design skills; keep anything only if it earns its context cost.
3. **Design system first** — one loop producing `docs/product/design-system.md` + the real `packages/ui` tokens/components (DSN-01). Claude Design's design-system import (or a `/design-sync` from a first exploration) may generate the first cut — gate it like any artifact. Every surface respec then references it, so the surfaces can't drift apart.
4. **Per surface: the screenshot loop** (DSN-02) — one surface = one [file 05](05-feature-loop.md) feature (landing, moderator console, participant surface, board, admin); because M1–M10 kept logic out of components, these loops touch only presentation. Iterate visually — in Claude Design's canvas where it helps, and always in the running app: `pnpm dev` → ⌘⇧4 screenshot → annotate in Preview/Markup → paste into Claude Code with change requests → repeat until the brief's adjectives hold on your phone *and* a laptop. Checklist step mandatory (a11y + responsive audit) before each surface merges.
Also in scope here, not earlier: **board appearance controls** (DSN-03 — high-contrast toggle, text/line sizing, responsive projector→TV→phone) and **event branding** (STD-10 — moderator logo + accent color, the only image in the product, gated = accountable; its single implementation home is this milestone's design system, per decision P3-7).

### 9.8 M12 — Operational maturity and compliance (`v1.0.0`)
- Start with measurement, not opinions: a loop for **observability** (OPS-01 — structured logs → Workers Logs/Logpush decision, DO/D1/KV usage dashboards, minimal alerting; keep-it-simple appetite) — you can't refactor for cost you can't see.
- Then the **constitution audit** (OPS-04): `Acting per Article III, audit the codebase for operator burden: dependency count/freshness, dead code, per-event DO storage growth, alarm/cron job health. Output a ranked refactor backlog.` → each accepted item becomes a small file-05 loop.
- **Retention verification, not introduction** (OPS-03 extension): the purge machinery has been running since M4 — here you verify it (evidence that every lifetime in the data-lifecycle table actually enforces, monitoring on purge-job health), never build it fresh. If verification finds gaps, they're bugs against the M4 spec.
- **Load-test the DO path** (OPS-04: scripted WS clients, ~500 participants, 1 event) before declaring stability — **degraded modes are designed only after this evidence** (decision T9-4), and the P1-era limits question (OQ6: participant/event limit *values* vs. the 1,000 ceiling — first pass runs pre-beta per the decision log; the counting model is fixed as concurrent presence, FR-2) gets its evidence-based second pass with the same data.
- Author the **rebuild-from-zero runbook** (OPS-05, `docs/runbooks/rebuild-from-zero.md`) — your requested emergency document: a loop whose spec is literally "starting from nothing but this repo, reach a live PRD". Its raw material is the `docs/runbooks/manual-config-log.md` you've kept since [file 02](02-setup.md) plus F000's `resource-bootstrap.md`; Claude drafts the runbook from log + workflows + wrangler config, and if you want it *proven*, walk it once against a scratch Cloudflare account. It also documents the **Cloudflare-native emergency levers** (WAF, Turnstile mode escalation) and the **admin-wall reset procedure** (decision T8-10). This is also the moment the Terraform/Pulumi question re-opens **with data** ([file 02 §2.8](02-setup.md) posture): count the runbook's remaining console-only lines — a handful means the posture holds; a page means evaluate the Cloudflare providers now, against their then-current coverage.
- Produce the **cost model** (OPS-02, `docs/ops/cost-model.md`): per-event and per-month formulas from *measured* usage (requests, DO duration, D1 row reads/writes, KV ops, emails) × current Cloudflare prices, with 2–3 worked scenarios (a 50-person meetup, a 500-person conference day, a slow month). This feeds the M6 admin cost panel's € projection, powers the **alert-only budget threshold** (notify, never auto-flip — decision T8-10), and turns "what does growth cost me" into arithmetic against the ~€20 busy-month target.
- The **compliance pack** (OPS-06): lightweight ROPA, breach-response runbook (72 h duty, delete-all-data as last resort, the Art.-34 notification procedure via admin export + external send — moderator accounts are the only *durable* data subjects; transient real-names event content within its 3-day window is in breach-assessment scope, FR-4), Cloudflare DPA check.
- Ship `v1.0.0` — the **first production-mature release**: complete product, visual system, operational controls, measured reliability, recovery, compliance foundation.

### 9.9 M13 — Monetization, if ever (PARKED · provisional `v1.1.0`)
Do nothing until you decide to earn money. When you do: kickoff ritual on the M13 epic; the billing provider (Stripe or a **Merchant-of-Record** like Paddle/Lemon Squeezy — an MoR handles EU VAT for you, which materially reduces the German paperwork; weigh in the interview) integrates as *one new grantor* writing rows into the M8 `grants` table (BIL-01) — if any other code needs changing, M8 violated Article VI and converge/fix that first. Two workstreams ride along, both non-coding: (1) the **German-operator legal pack** (BIL-02 — Impressum/DDG, USt/Kleinunternehmerregelung, AGB, Widerruf, invoicing) resolved with a Steuerberater/lawyer before charging anyone; (2) the **compensation & communication policy** toward paying customers for drastic admin controls (emergency stop, lockdown — decision T8-4), plus billing's own legally mandated transactional email set. The provisional `v1.1.0` moves if other post-`v1.0.0` releases land first.
