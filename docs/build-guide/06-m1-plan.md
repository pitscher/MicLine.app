### F000 — Foundations & delivery pipeline *(reduced ceremony: skip clarify-interview depth, skip checklist — but its plan must read [file 03](03-github-operations.md), which is the design F000 implements)*
Scope, in four bundles:
1. **Monorepo & app skeleton:** pnpm/turbo workspace; a committed `.node-version` (`24`) + the exact `"packageManager": "pnpm@11.⟨x.y⟩"` pin in the root package.json — the two lines that make the repo, not the machine, own runtime versions ([file 02 §2.1/§2.9](02-setup.md)); `apps/web` (React Router framework mode, major per the plan addendum below, + `@cloudflare/vite-plugin` + Tailwind v4 + Hono mounted for `/api`,`/ws`); `packages/shared` (zod, WS contracts, Drizzle schema placeholders) + `packages/ui` (design tokens — M4 demand); Biome; Vitest + `@cloudflare/vitest-pool-workers` smoke test; strict tsconfig; Hono baseline middleware for **security headers** (HSTS in prod, `X-Content-Type-Options: nosniff`, `frame-ancestors 'none'`, CSP in report-only — tightened in F007); **i18n plumbing: Lingui** with locales `en`+`de` — every user-facing string via its macros from the first component, `pnpm i18n:extract`, catalogs compiled inside `build` so an unextracted string fails CI (verify Lingui's current integration with the chosen React Router major + Vite during planning).
2. **Environments ([file 03 §B](03-github-operations.md)):** `wrangler.jsonc` with `env.dev` (worker `micline-dev`, custom-domain route `dev.micline.app` — wrangler creates the DNS record on first deploy — its own D1 database + KV namespace) and `env.production` (worker `micline`, route `micline.app`, separate resources). Names/ids only — never secrets. `.dev.vars.example` documents required local keys with no values. Plus, per the IaC posture ([file 02 §2.8](02-setup.md)): a committed **`docs/runbooks/resource-bootstrap.md`** holding the exact `pnpm wrangler d1 create …` / `pnpm wrangler kv namespace create …` commands per environment and where each returned id goes in `wrangler.jsonc` — the repo alone must be able to recreate every Cloudflare resource.
3. **GitHub plumbing (file 03 §A/§C/§D):** `ci.yml` with five named checks `lint` `typecheck` `test` `build` `secrets` (gitleaks) — workflows take Node from `.node-version` (setup-node's `node-version-file`) and pnpm from the `packageManager` pin, so CI and laptop can't drift; `deploy-dev.yml` (push to main → GitHub Environment `dev` → `wrangler d1 migrations apply --env dev --remote` → `wrangler deploy --env dev`); `deploy-prd.yml` (on `release: published` → GitHub Environment `production` **with you as required reviewer** → migrate + deploy `--env production`); `.github/release.yml` (auto release notes categorized by `type:*` labels); issue templates (feature_request, bug_report) + `config.yml` (blank issues off, questions → Discussions) + PR template (purpose, `Closes #issue`, self-review checklist, and an **AI-assisted change** disclosure checkbox naming the tool/model — the EmDash pattern, [file 03 §A](03-github-operations.md)); `SECURITY.md`; workflow hardening per [file 03 §G](03-github-operations.md) — third-party actions pinned to commit SHAs, top-level `permissions: contents: read`; committed pre-commit gitleaks hook (replaces the throwaway from [file 02 §2.4](02-setup.md)); Dependabot (npm + actions, grouped weekly); CodeQL default setup; README badges (CI, release, license) + Roadmap/Changelog links.
4. **Agent plumbing:** nested `CLAUDE.md` stubs in `apps/web/worker/` and `packages/shared/` ([file 08 §B](08-claude-code-reference.md)); if you adopted the typescript-lsp plugin (file 08 §C), install its prerequisite: `npm i -g typescript-language-server typescript`.
Plan addendum: `Read docs/build-guide/03-github-operations.md as a binding design input. Fetch developers.cloudflare.com/workers/llms.txt and the @cloudflare/vite-plugin + React Router framework-mode docs; decide the React Router major here against what the current official Cloudflare template targets (v8 is current as of 2026-07; tech-decisions pinned v7 at planning time) and record the choice in tech-context.md; scaffold with the current official template, do not hand-roll config from memory. Wrangler environments: dev and production exactly as specified; document the local + per-env migration commands in the repo README.`
**F000 exit checklist:** (a) `pnpm dev` serves a hello page through the Worker locally; (b) CI green on the F000 PR; (c) a deliberately planted fake secret is blocked by the pre-commit hook; (d) now do [file 02 §2.7 row 1](02-setup.md) — Workers Paid, CI token, the GitHub `dev` + `production` environments with their secrets — then re-run `deploy-dev.yml`; its first success also *creates* the `dev.micline.app` DNS record (custom-domain route), so confirm the hello page loads there; (e) turn on branch protection (require the five checks; PRs; linear history) — last, so F000's own merge wasn't blocked by not-yet-existing checks.

### F003 — EventRoom realtime + question line *(the heart; Opus 4.8, EFFORT max for plan)*
Scope: tech-context rules 2 & 4 in full — DO with SQLite storage via Drizzle `durable-sqlite`; WebSocket **Hibernation API** with role/sid tags; edge-authenticated role headers; complete status state machine + invariants; fractional-index ordering + renormalization; vote dedupe + self-upvote rejection; snapshot/delta protocol + role-based visibility filtering; end→stats snapshot. Tests: every state-machine transition, one-speaking invariant, reorder, dedupe, visibility — the ≥80% service/DO coverage gate starts here.
Plan addendum: `Fetch developers.cloudflare.com/durable-objects/llms.txt — specifically WebSocket hibernation, SQLite storage API, and alarms. Mental model note for the operator: a DO is a single-threaded reliable actor (cf. Azure Durable Entities); all mutations serialize through it, which is WHY the invariants hold — the plan must not introduce any state outside the DO. WS message contracts are zod schemas in packages/shared, shared verbatim by client and DO. Enforce an Origin allowlist on the WS upgrade (micline.app, dev.micline.app, localhost) before any auth work. Split tasks into Phase A (schema+state machine+services+unit tests, no sockets) and Phase B (WS transport, snapshot/deltas, integration test driving two scripted clients through create→submit→upvote→enqueue→call_to_mic→answered).`
Implement in two runs (Phase A, gate, Phase B) per file 05 §7.8.

### Optional tooling loop — translation status board (Lunaria) *(not part of the M1 gate; slot it anywhere after F001, ideally before F004/F005 when UI strings start multiplying)*

You wanted a translation dashboard "like lunaria.dev, updated by a GitHub Action, from early days" — and asked whether the guide sets it up or Spec Kit does. Answer: **run it as a light-ceremony Spec Kit loop** (like F007) — it touches CI and committed tooling, so a one-page brief + short loop keeps it governed; it is *not* bootstrap work and *not* an M1 feature, so it never blocks the release gate. What baseline you already have without it: F000's build fails CI on any unextracted string, and `pnpm lingui extract` prints per-locale missing counts on demand.

Verified facts to plan against (as of 2026-07-05 — re-verify in the loop, it's **public beta**): Lunaria tracks translation status purely from **git history** (source file newer than its localized sibling ⇒ "outdated"); `lunaria build` emits a **static dashboard** (`dist/lunaria/index.html`); the official **GitHub Action** (`yanthomasdev/lunaria-action`) comments on PRs with the status impact of the change. One catch for this stack: Lunaria's key-level `dictionary` type supports json/yml/md/js/ts — **not Lingui's default `.po` catalogs** — so either (a) track the catalogs as `universal` type (file-level done/outdated per locale; simplest), or (b) switch Lingui's catalog format to JSON if you want per-key missing lists. Dashboard hosting: GitHub Pages is fine for tooling ([file 03 §A](03-github-operations.md) matrix) — or skip hosting entirely and let the Action's PR comments be the dashboard; solo, that's often enough.

Brief seed (paste into a `docs/product/feature-briefs/` file, then run the loop): *"Contributors and I can see, per locale, which catalogs are done/outdated/missing — on every PR (Action comment) and optionally on a static dashboard — without any external service."* Plan addendum: `Fetch lunaria.dev docs (getting-started, configuration, github-action) fresh — the tool is beta. Decide universal-vs-dictionary tracking against our Lingui catalog format and record it in the plan. The Action must not become a required check (informational only). If a dashboard is hosted, it deploys from CI, never by hand.` When M2's rented TMS candidate lands ([file 07 §9.1](07-m2-to-m5.md)), it either replaces this or feeds it — decide then.

### M1 release gate (run after F007 merges) — do these yourself, in order
1. Secret hygiene: `gitleaks git` over the full history (`gitleaks detect` is its deprecated alias); confirm secret scanning + push protection + private vulnerability reporting are ON; `git grep -iE 'secret|api[_-]?key|token' -- ':!*.md'` returns nothing suspicious.
2. Confirm every [file 02 §2.7](02-setup.md) row is done (per-environment Cloudflare secrets in **both** GitHub environments, email domain onboarded, Turnstile covers both hostnames).
3. Full two-device smoke on **DEV** (dev.micline.app): create → join via QR → submit → upvote → enqueue → call-to-mic → "You're up!" → answered → end event → recap. Real magic-link email arrives; headers show SPF/DKIM/DMARC = pass.
4. Ship: `gh release create v1.0.0 --generate-notes --title "MicLine v1.0"` → edit the generated notes into a human changelog → publishing triggers `deploy-prd.yml` → open the Actions run and **approve the `production` environment** (your deliberate go-button) → deploy runs.
5. Repeat the smoke test on **PRD** (micline.app).
6. Go public: post a welcome thread in Discussions, verify README badges + Roadmap/Changelog links resolve, pin the roadmap Project. Take a break — you shipped, and the Releases page now tells users exactly what v1.0.0 contains.
