## §8 M1–M4 execution plan (milestone by milestone)

**Authority first:** `docs/product/feature-inventory.md` owns WHAT ships in which milestone and under which target version — this file owns HOW that work executes (order, plan addenda, gates). The **F-numbers below are engineering work packages**, not product features: F000–F007 are the legacy set the inventory itself cites as provenance ("(F000)", "(F003)" …), now mapped onto M1–M4; F008–F013 are **provisional** packages for the scope P2/P3 added. The tech interview ([file 04 §4](04-product-definition.md)) proposes the final brief cut per milestone and **Gate 2 fixes it** — if it re-cuts or renumbers, the inventory IDs in the map below are the stable reference, and PROGRESS.md gets the new names.

**Every milestone ends in a release.** Completing a milestone = publish its target version (`gh release create v0.⟨N⟩.0 …`) → approve the `production` environment → smoke PRD. `micline.app` therefore runs real code from `v0.1.0` on, but stays **unannounced** — no links, no posts — until the M4 gate makes it the first responsible public beta. Mid-milestone increments that you want on PRD early ship as prereleases of the milestone's target (`v0.4.0-beta.1`); the SemVer policy lives in [file 03 §D](03-github-operations.md).

### §8.1 M1 — Deployable foundation (`v0.1.0`)

#### F000 — Foundations & delivery pipeline *(reduced ceremony: skip clarify-interview depth, skip checklist — but its plan must read [file 03](03-github-operations.md), which is the design F000 implements)*

Plan addendum: `Read docs/build-guide/03-github-operations.md as a binding design input. Fetch developers.cloudflare.com/workers/llms.txt and the @cloudflare/vite-plugin + React Router framework-mode docs; decide the React Router major here against what the current official Cloudflare template targets (v8 is current as of 2026-07; tech-decisions pinned v7 at planning time) and record the choice in tech-context.md; scaffold with the current official template, do not hand-roll config from memory. ### §8.3 M3 — Open-line event runtime (`v0.3.0`)

#### F003 — EventRoom realtime + the open line *(the heart; Opus 4.8, EFFORT max for plan)*
Scope, per LINE-01…08 + MOD-02: tech-context rules 2 & 4 in full — one DO per event with SQLite storage via Drizzle `durable-sqlite`; WebSocket **Hibernation API** with role/sid tags; edge-authenticated role headers; the **open-line state machine** — submit → appended to the public FIFO line (order = arrival; **your position only ever improves**), Now speaking pinned, answered (next auto-pulled) / **skip with stub semantics** (text vanishes from all public surfaces instantly; an attribution + "skipped" stub stays in the participant line; the board never shows skipped items; the owner keeps an honest chip) / withdraw (slot frees instantly) / ~10 s single-level undo for answered and skip; **one active waiting item per participant** (LINE-08 — session-scoped, fixed platform rule; no fingerprinting or IP binding, ever: decision T1-2); **event-full enforcement against the concurrent-presence participant cap** (FR-2 — a participant is a currently-active session; leave/disconnect frees the slot; staleness semantics per the tech interview); **announcements** (MOD-02: one current announcement, replaces the previous, bounded display lifetime from platform config with uniform auto-expiry on every surface — a DO alarm fits — subtle draining cue on the board, clear-early action, re-send restarts); snapshot+deltas protocol + role-based visibility filtering; graceful reconnect; end→ended state. **The domain model is built two-mode-shaped from day one** (decision B10): statuses and schema must extend to pool/review/declined/merged (M9) without migration pain — but no pool, no votes, no review, no reorder machinery ships now; the M3 line is append-plus-removals only. Tests: every state-machine transition, the one-speaking and one-active-item invariants, skip-stub visibility, undo windows, announcement expiry — the ≥80% service/DO coverage gate starts here. **Binding acceptance criteria (decision T9-2, verbatim in the spec):** ① no silent loss — a submission is durably saved and visible, or explicitly reported failed; ② retries never create duplicates; ③ explicit save states (sending / saved as #N / failed, tap to retry); ④ automatic recovery — identity, own items, chips, and position restore after reload/reconnect with no user action; "please refresh" is never the designed path.
Plan addendum: `Fetch developers.cloudflare.com/durable-objects/llms.txt — specifically WebSocket hibernation, SQLite storage API, and alarms. NOTE: a DO has exactly ONE alarm — the EventRoom's timed concerns (announcement expiry, auto-open/auto-end, purge timing) must multiplex it or move to cron, per the tech interview's scheduling decision; the plan must state which. Mental model note for the operator: a DO is a single-threaded reliable actor (cf. Azure Durable Entities); all mutations serialize through it, which is WHY the invariants hold — the plan must not introduce any state outside the DO. WS message contracts are zod schemas in packages/shared, shared verbatim by client and DO. Enforce an Origin allowlist on the WS upgrade (micline.app, dev.micline.app, localhost) before any auth work. Ordering: arrival order is authoritative in open mode; design the schema so M9's curated ordering slots in without a rewrite, but build no reorder machinery now. Split tasks into Phase A (schema+state machine+services+unit tests, no sockets) and Phase B (WS transport, snapshot/deltas, integration test driving two scripted clients through create→open→join→submit→saved-as-#N→second-submit-blocked→skip+stub→undo→answered-auto-pull→you're-up→withdraw→announcement-expiry→reconnect-snapshot→end).`
Implement in two runs (Phase A, gate, Phase B) per file 05 §7.8.
