## A — Audience & positioning (P1)

- **A1 · Live-mic model is the identity.** The line orders *people who speak* (walk-up, mic passed, or unmuted); "up next" prepares people mentally. Why: it's the wedge no incumbent owns and matches the name. Rejected: read-aloud-only model (Slido's home turf; falls out for free anyway).
- **A2 · Primary segment: in-room events, meetup-to-conference (≈30–500).** Classrooms tolerated, not marketed; livestreams out of design scope M1/M2 (no OBS/chat integrations). Why: each other segment drags hidden cost onto a solo operator. **[AMENDED P2 → T2-7]:** online/hybrid meetings are a fully supported usage context (URL join works in a video call); in-room stays the design center; the integration rejection is untouched.
- **A3 · One event = one room/session.** Parallel tracks = separate events; no umbrella/series concept. Design envelope ~500 concurrent participants, platform hard cap 1,000 (see G1).
- **A4 · Buyer = prosumer moderator, self-serve.** B2B = a bigger invite code sold manually via email. Rejected: teams/seats/org accounts, white-label.
- **A5 · Pillar order: the line → zero friction → privacy; simplicity is a principle, not copy.** Open source is not a marketing pillar: the running SaaS must not link/refer to the repo until the operator changes course.

# P2 — Final product interview (2026-07-12)

## P2-1 — M1 scope locks & line semantics

- **T1-1 · One active waiting line item per participant (open line), fixed platform rule — new feature LINE-08.** Answered/skipped/withdrawn frees the slot; then the participant may submit again. Why: one person can't stand in a physical mic line twice; multiple positions dilute the fairness pitch on a public FIFO board; the own-question chip is singular-shaped. Rejected: multiple active items in M1 (fairness dilution, forces a "my questions" list view); per-event configurability in M1 (curated mode gets its own limit, T6-1).
- **T1-2 · Enforcement is session-scoped; device fingerprinting and IP-identity binding REJECTED, even event-scoped.** The leave-and-rejoin residual is accepted as best-effort (same stance as vote dedupe); marketing/privacy copy claims exactly "enforced per session". Why rejected: venue NAT makes IP useless at in-room events; fingerprint collisions on uniform phone fleets block innocent participants with no support path; fingerprint+IP is personal data with TDDDG-consent exposure (consent UI = the banner pillar 2 forbids) and kills the "participants: nothing to delete" privacy line; the loophole appends at the tail of a visible line and is moderator-managed per D5. Rejected alternatives: operator's event-scoped fingerprint+IP proposal (withdrawn after challenge); anonymous cookie surviving "Leave" (betrays the explicit reset action and the B6 typo escape hatch). Recorded explicitly so the tech interview cannot reintroduce tracking to close the gap.
- **T1-3 · Every line item requires written text (platform 1–280); request-to-speak REJECTED for M1 and PARKED (PRK-11).** Why: a board of "wants to speak" entries destroys the line's information value; in nickname mode it tells the room nothing; skip loses its content basis; a second item type ripples through board, review, filter, export. Zero-cost workaround: type "I'd like to comment on X". Unparking trigger: real-event demand.
- **T1-4 · Withdrawal unchanged (LINE-05);** slot frees instantly; nothing else needed under one-active-item.
- **T1-5 · No intake-closed state in M1; intake close ships M2A (MOD-05) as a manual, freely reversible toggle.** M1 lifecycle stays scheduled → live → ended (full/unknown are join screens). M1 workaround: announcement + manual end; the transparent line length self-calibrates expectations. Rejected: M1 inclusion (not essential for a first event); timers/auto-close (REJ-07 — surprise closure violates no-surprise moderation).
- **T1-6 · Per-event min/max question length within the platform 1–280 rule** — moderator creation setting, defaults = platform bounds; validation/copy details at tech interview / F001 spec. [ASSUMED M1, operator-approved.]

## P2-2 — Event information, primer, announcements, content richness

- **T2-1 · M1 primer stays pre-join-only plain text;** the announcement is the only mid-event channel in M1.
- **T2-2 · CAND-01 resolved as "reuse the primer":** in M2B the primer becomes re-openable in-event (ⓘ sheet on the participant surface), folded into STD-08. A separate information panel is REJECTED (REJ-03 — second content container, double maintenance, settings-freeze breach). Canonical division: **primer = static reference (frozen at start), announcement = live channel.**
- **T2-3 · Links render inert in all current milestones** — moderator-authored content is never auto-linked or tappable; URL-shaped text tolerated but inert. Why: zero-gate + QR entry makes MicLine an ideal quishing loader; a solo operator can't run link scanning or URL abuse triage. *Clickable links* are PARKED (PRK-09), not permanently rejected — operator explicitly kept the future option; unparking requires a dedicated security/abuse review.
- **T2-4 · No images in moderator content in current scope;** *content images* PARKED (PRK-10) with storage/purge, EXIF-privacy, and content-moderation-liability review as unparking conditions. The M2B branding logo (STD-10) is deliberately the only image in the product — narrow, single-purpose, entitlement-gated (accountable uploader).
- **T2-5 · Primer never renders on the projector board.** Announcement display model (amends B18/MOD-02): bounded lifetime (platform config value, direction ~60 s), **uniform auto-expiry on board and participant surfaces** (no stale messages on unattended machines; one mental model), subtle draining progress cue on the board [ASSUMED style — numeric countdown available on veto], clear-early action, re-send restarts the lifetime. Why uniform: split lifetimes create two mental models and stale phone banners; a stale announcement presented as current is worse than a missed one. Rejected: board-only expiry; persistent-until-replaced.
- **T2-6 · Acknowledgment modes sharpened (amends PRT-05, M1):** skippable vs required explicit confirmation; in required mode the moderator customizes the confirmation statement (plain text, capped; localized default "I understand."). Purpose: a poke to actually read — explicitly not bulletproof. Boundary derived from data minimalism: **the acknowledgment is a gate, not a record** — MicLine never stores who confirmed what; it is not legal proof of consent.
- **T2-7 · Online-use positioning clarification (amends A2 wording, not scope):** in-room = design center; online/hybrid meetings = fully supported usage context; broadcast/livestream tooling stays out of scope. "Specific in focus, deliberately versatile."

## P2-3 — Projection safety, board behavior, live-event control

- **T3-1 · Skip-stub model (amends B13):** skipping removes the question **text** from all public surfaces immediately; a minimal stub (attribution + "skipped") remains in the participant line; the projector board never shows skipped items; the owner keeps an honest chip. Why: transparency means the room sees *that* something was removed, not that the content stays amplified. Rejected: full text remains (frames the abuse); silent disappearance (violates transparent moderation).
- **T3-2 · Moderator board-hide exists, M2A (MOD-06).** Why not M1: with the stub model, skip kills amplification in seconds; first events are low-risk. Why it exists: coordinated waves mean whack-a-mole on the projector; non-abuse uses (break, alarm, sensitive moment). Pairs with intake close as the live-event-controls cluster.
- **T3-3 · Hide board is board-only.** Participant surfaces stay fully live (line, chips, withdraw, submissions); intake continues. Why: the harm is public amplification; killing phones makes the product vanish for the room; content that must leave phones is what skip is for.
- **T3-4 · Hidden board shows event title + neutral message + wordmark; no line content, no QR/code.** The message has a system default and is **moderator-customizable at hide-time** (live control per the B18 precedent — a hide-message can't be a frozen creation setting; platform max-char cap). Tech requirement logged: **the board URL must never expose the event ID or join code** (own capability token; not enumerable). Styling/animation at spec.
- **T3-5 · Naming: EN "Hide board" / DE "Board verbergen" (term sheet); explicitly distinct from the admin emergency stop.** **No audit-log entry** for hide/unhide — ops logs only (short-lived). Why: the audit log is for privileged admin/entitlement actions; no moderator in-event action keeps a durable trail (REJ-09); a 12-month trail of one moderator action would be inconsistent and privacy-hostile. [Initial audit proposal withdrawn by Claude after review; operator confirmed.]
- **T3-6 · Locale derivation model [PARTIALLY SUPERSEDES H2]:** console = moderator device default, adjustable anytime, persisted on account · event language = moderator device default at creation, adjustable before start only · participant surface = **participant device default**, adjustable anytime, persisted in session cookie/device (participants have no accounts — the "account" phrasing in the proposal was corrected) · board = event-language default, adjustable per viewer, persisted per device until event end · emails = moderator's console language. UI chrome only, never user content. Named spec-time edges: register-variant resolution (direction: event variant wins when base languages match); one person holding participant surface + board (independent persistence).

## P2-4 — M2 line modes, review, duplicates, upvotes (closes OQ2)

- **T4-1 · Curated mode public shape.** Two zones on the participant surface: the line (committed order) and the pool. **Pool visibility is a per-event creation setting:** hidden pool auto-disables upvotes; visible pool lets the moderator toggle upvotes; both freeze at start. The owner always sees their own "in pool" chip (no personal black hole). Participant pool sort: votes-desc when upvotes on, newest when off; console sorts freely. Moderator actions: place/insert, drag-reorder, return-to-pool, skip (stub semantics in both zones). Unselected items simply remain in the pool. Position stability documented as an open-line-only guarantee. Defaults [ASSUMED → M2B kickoff]: pool visible, upvotes on. Noted interplay: with a hidden pool, review approval has no participant-facing effect — review's remaining value is the explicit decline flow.
- **T4-2 · Review semantics.** Open+review: approval **appends to the line at approval time** — approval order is line order. Rejected: preserving original submission order (retro-insertion above committed items breaks position stability); manual placement in open mode. Curated+review: approval releases into the pool; line placement stays a separate curation act. Declined: owner-visible "declined" chip **with optional short moderator reason** (plain text, capped, inert, dies with purge); invisible to everyone else forever **[scoped FR-4:** = never shown to other participants or the board; the moderator, co-moderators, and every export see declined items**]**; reversible until event end; retained until purge. Rejected: silent limbo (black hole).
- **T4-3 · Duplicate merge (pool-only).** Moderator selects items, picks a **primary**; primary's text represents the group with an "×N" badge that travels with it (pool and line — it legitimizes curation, it isn't a vote count); merged items fold away publicly; owners' chips show "merged" and inherit the primary's outcome; votes combine with one-voice-per-participant dedupe (best-effort); unmerge until consumed. Stated plainly: the primary's submitter gets the mic. No merge machinery in open mode — the review tray declines duplicates there. Rejected: merging line items (strips committed positions).
- **T4-4 · Upvote boundaries.** Curated mode only, pool items only; no vote UI or counts on line items on participant surfaces/board (console may show counts); nameless in both name modes; session-deduped best-effort. Invariant verbatim: **votes advise the human, never reorder the line.** No votes in open mode (B11 stands).
- **T4-5 · No review↔co-moderator dependency.** Solo review stays possible; the creation-time consequence hint ("on = you're the bottleneck") is the guardrail. Force-rank order unchanged. Rejected: hard dependency (paternalistic, tangles ship order).

## P2-5 — Moderator console efficiency, notes, follow-up, filters

- **T5-1 · Follow-up flag: yes (MOD-08, M2A with export).** Moderator-only boolean on any item, set during event or recap window; never participant-visible, never affects public state; export column; purge-bound.
